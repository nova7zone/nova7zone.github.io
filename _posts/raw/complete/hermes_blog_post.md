# 빈 ARM 서버에 24시간 AI 비서 심기 — Hermes + OpenRouter + Telegram 구축기

> OCI A1(ARM) 인스턴스에 Nous Research의 Hermes Agent를 올리고, OpenRouter 무료 모델을 붙여, 폰에서 텔레그램으로 대화하는 비서를 만들기까지. 명령어보다 **왜 그렇게 골랐는지**에 무게를 둔 기록.

---

## 0. 무엇을 만들려고 했나

목표는 단순했다. **항상 켜져 있는 서버에 AI 에이전트를 상주시키고, 폰에서 텔레그램으로 부려먹는 것.** 거창한 자동매매 위임 같은 게 아니다. 간밤 증시 요약을 아침에 받고, 서버 로그를 감시하고, 외출 중 간단한 리서치를 시키는 — 딱 "운영 비서" 수준.

선택한 조합은 이렇다.

- **호스트:** Oracle Cloud의 A1 Flex 인스턴스 (Ampere ARM / aarch64, 4코어 / 23GB RAM)
- **에이전트:** Hermes Agent (Nous Research, MIT 라이선스, 오픈소스)
- **두뇌(LLM):** OpenRouter 경유 무료 모델
- **인터페이스:** Telegram 봇

핵심 전제 하나를 먼저 못 박았다. **A1은 GPU 없는 CPU 전용 ARM 박스다.** 이 사실이 뒤에 나올 "로컬 LLM이냐 API냐" 결정을 통째로 좌우했다.

---

## 1. 먼저 정리한 역할 분담

설치에 손대기 전에 경계선부터 그었다. 이게 없으면 "AI한테 다 시키자"는 유혹에 빠지기 쉽다.

- **Claude.ai = 머리 쓰는 일의 사고 파트너** — 전략 로직, 코드 설계, 무거운 판단.
- **Hermes = 운영자** — 감시, 수집, 스케줄, 알림.

그리고 절대 넘지 않을 선:

> 매매 판단·주문 실행은 Hermes에 위임하지 않는다. 모니터링·리서치까지만. **결정과 실행은 사람이 쥔다.**

약한 모델에 비가역적·금전적 행동을 맡기지 않는다는 원칙이다. 이 선을 먼저 그어두니, 이후 모든 설정 단계에서 "이건 켜도 되나?"의 판단 기준이 명확해졌다.

## 2. 설치 — 빈 서버에서 시작

### 의존성

```bash
sudo apt update
sudo apt install -y curl git build-essential
```

`build-essential`이 중요하다. aarch64는 일부 Python 휠에 ARM 프리빌드가 없어서, 없으면 설치 중 소스 빌드가 일어나고 컴파일러가 없으면 거기서 깨진다. (OCI Ubuntu 이미지는 이미 셋 다 갖고 있어서 실제로 새로 깔린 건 없었다.)

### 가상환경은 직접 만들 필요 없다

처음 헷갈렸던 지점. "가상환경에 깔아야 하나?" — 답은 **"내가 만들 필요 없다"**였다. Hermes 설치 스크립트는 `uv` 기반이라, 의존성·저장소 클론·**가상환경·전역 명령 설정까지 자동으로** 처리한다. venv를 손으로 만들면 오히려 충돌한다. 가상환경을 만들지 *말지*가 아니라 *누가 만드느냐*의 문제였고, 여기선 도구가 알아서 만든다.

### 원라인 설치

```bash
curl -fsSL https://hermes-agent.nousresearch.com/install.sh | bash
source ~/.bashrc
hermes doctor
```

`curl | bash`는 내용을 못 보고 실행하는 방식이라, 불안하면 먼저 받아서 `less`로 훑고 돌리는 게 안전하다. 그리고 **`source ~/.bashrc`를 빼먹으면** `hermes` 명령을 못 찾는다 — PATH 미반영, 가장 흔한 첫 함정이다.

> 참고: 혼자 쓰는 비서 호스트라면 root가 아니라 **일반 사용자(ubuntu)로 설치**하는 게 맞다. `~/.hermes/`에 격리되고, 나중에 systemd 서비스도 같은 사용자로 일관되게 잡힌다.

## 3. 모델 백엔드 — 결정의 연속

### 로컬 LLM? 이 박스에선 비추천

23GB RAM이면 모델을 *올리는* 데는 넉넉하다. 하지만 속도 병목은 RAM이 아니라 **연산**이다. CPU 전용 ARM 4코어에서 소형 모델은 초당 몇 토큰 수준 — 도구 호출을 수십 번 엮는 에이전트 루프엔 너무 느리다. 로컬의 유일한 진짜 보상은 프라이버시인데, 이번 건은 그게 중요하지 않았다. **결론: API로 간다.**

### OpenRouter vs Nous Portal

설치 마법사가 두 길을 제시했다. 기본값은 Nous Portal(OAuth 로그인, 키 없이 즉시)이었지만, 이건 별도 구독 과금 구조였다(Plus 기준 월 $20). 우리 용도(데이터 수집+요약)는 무료 모델로 충분하니, **월 구독은 과지출**이었다.

그래서 OpenRouter를 골랐다. 무료로 시작하고, 필요하면 소액 충전으로 한도를 올리는 종량제. 결정적으로 — **나중에 Portal이 필요해지면 `hermes model` 한 줄로 갈아탈 수 있다.** 모델/공급자는 "두뇌 교체"라, 메모리·스킬·cron·게이트웨이 설정은 그대로 남는다. 잘못 고른 게 아니라 *되돌릴 수 있는* 결정이었기에 부담 없이 OpenRouter로 출발했다.

### 모델 선택의 진짜 기준: 도구 호출

무료 모델 목록에서 중요한 건 "이름값"이 아니라 **도구 호출(tool calling)이 안정적이냐**다. 에이전트는 결국 도구를 불러서 실제로 뭔가 해야 하니까. "Nous가 만든 모델이니 Nous Portal에서 그걸 쓰면 되겠지"는 오답 — 그 모델군은 채팅·추론용이라 빠른 도구 호출 루프엔 부적합하다는 안내가 있었다.

최종 선택: **`nvidia/nemotron-3-super-120b-a12b:free`** — 명시적으로 agentic 용도로 설계됐고, 목록에 있어 커스텀 입력 오타 위험도 없고, 무료다.

## 4. 안전 설정 — Local 백엔드의 핵심

터미널 백엔드는 `Local`을 골랐다. Docker 격리가 정석이지만, 이 비서의 본업(호스트 cron·로그 감시, 추후 다른 서버 SSH 접속)을 컨테이너에 가두면 첫날부터 디버깅 지옥에 빠진다. 대신 — **진짜 안전 경계는 백엔드 격리가 아니라 "승인·권한"에 있다.**

설치 후 설정 파일(`~/.hermes/config.yaml`)을 직접 확인했더니 기본값이 보수적으로 잘 잡혀 있었다.

```yaml
approvals:
  mode: manual          # 위험 명령은 항상 사람 승인
  timeout: 60           # 60초 내 응답 없으면 거부 (fail-closed)
  cron_mode: deny       # cron이 위험 명령 만나면 자동 거부
  destructive_slash_confirm: true
```

특히 `cron_mode: deny`가 마음에 들었다. 새벽에 자동으로 도는 작업이 위험 동작을 *자동 승인*해버리면 곤란하니까. 사람이 없으면 그냥 거부 — 방어적으로 옳다.

도구 선택에서는 안 쓰는 것들을 정리했다. **macOS Computer Use는 무조건 꺼야 한다** — 헤드리스 ARM 리눅스에선 작동도 안 한다. 이미지 생성·TTS·브라우저 자동화도 일단 껐다(불필요 + 잠재 과금/위험). Terminal·Code·File은 위험하지만 본업 필수라 켜두되, manual 승인 모드로 보호한다.

## 5. 검색 공급자 — 옛 정보에 데인 지점

여기서 한 번 헛짚었다. 처음엔 "Brave Search 무료 티어(월 2,000쿼리)"를 추천했는데, **확인해보니 2026년 2월에 신규 사용자 무료 티어가 폐지**됐다. 지금은 $5 선불 크레딧 + 카드 필수 + 지출 상한 없음. cron이 자동으로 도는 작업에 "지출 상한 없음"은 위험 요소다.

> 교훈: 가격·티어 같은 건 *지금 시점*을 확인해야 한다. 마법사의 `(Free)` 라벨조차 옛 정보를 들고 있을 수 있다.

그래서 **DuckDuckGo(ddgs)**로 갔다. 키 불필요, 카드 불필요, 즉시 작동. 비공식 스크래핑 기반이라 가끔 불안정할 수 있지만, "헤드라인 수집·요약"엔 충분하다. 안정성이 정말 필요해지면 그때 유료나 셀프호스팅으로 올리면 된다.

## 6. 텔레그램 게이트웨이 — 가장 중요한 보안 단계

폰에서 쓰려면 텔레그램 봇이 필요하다. `@BotFather`로 봇을 만들고 토큰을 받은 뒤, `hermes gateway setup`으로 연결했다.

> 토큰은 봇을 조종하는 열쇠다. API 키와 동급의 민감 정보. 유출되면 BotFather `/revoke`로 재발급.

**전체 설치에서 보안상 제일 중요한 지점이 여기다.** 실제 보고된 사고가 있다 — 허용목록 없이 게이트웨이를 열어둬서, 봇을 발견한 아무 계정이나 DM을 보낼 수 있었고 봇이 정상 응답해버린 사례. 우리 봇은 **터미널 명령 실행이 가능한** 봇이다. 외부인이 붙으면 그건 곧 *서버에서 명령을 돌릴 수 있다*는 뜻이다.

그래서 철칙:

> 터미널 접근이 있는 봇에는 **절대 `GATEWAY_ALLOW_ALL_USERS=true`를 켜지 않는다.**

본인 텔레그램 User ID(`@userinfobot`으로 확인)만 허용목록에 넣었다. 허용목록에 없는 ID는 봇이 무응답으로 무시한다. 이게 외부인 차단의 전부다.

```
Allowed user IDs: 6348086360
✓ Telegram allowlist configured - only listed users can use the bot
```

Home Channel(cron 결과·알림이 도착할 곳)도 본인 ID로 잡았다. 모닝 브리핑이 본인 DM으로 오게.

## 7. 상시 가동 — System service로

게이트웨이를 어떻게 띄울지 물었을 때, 기본값은 `User service`였다. 하지만 이건 **SSH 로그아웃 시 꺼질 수 있고 부팅 시 자동 시작도 안 된다.** 24시간 비서엔 치명적이다. SSH 끊었다고 비서가 죽으면 말이 안 된다.

그래서 **System service**를 골랐다 — 부팅 자동 시작, 단 실행은 ubuntu 권한으로 제한.

여기서 또 하나의 함정. `sudo hermes ...`가 `command not found`로 깨졌다. 원인은 PATH — `hermes`는 `~/.local/bin`에 있는데 sudo는 root의 PATH를 쓰니 못 찾는다. 해결은 전체 경로 명시:

```bash
sudo /home/ubuntu/.local/bin/hermes gateway install --system --run-as-user ubuntu
```

`--run-as-user ubuntu`가 핵심이다. 부팅 자동 시작(시스템 서비스의 장점)은 얻으면서, 실제 실행 권한은 ubuntu로 묶어 권한 상승을 막는다.

```
✓ System service installed and enabled!
✓ System service started
Configured to run as: ubuntu
```

## 8. 첫 교신 — 폰에서

마지막 관문. 텔레그램에서 봇을 찾아 `/start`와 인사를 보냈더니, 몇 초 만에 응답이 왔다.

```
나: /start
나: 하이^^
Hermes AI: 안녕하세요! 😊 어떻게 도와드릴까요?
```

이 한 줄이 전 경로의 작동을 증명한다 — **폰 → 텔레그램 → A1 게이트웨이 → OpenRouter(Nemotron) → 다시 폰.** 그리고 본인 계정이 허용목록을 정상 통과했다는 뜻이기도 하다.

## 10. 배운 것 / 다음 단계

설치 과정에서 반복해서 확인한 원칙들:

1. **되돌릴 수 있는 결정엔 빨리 출발한다.** 모델·공급자는 한 줄로 교체되니, OpenRouter로 가볍게 시작.
2. **안전 경계는 도구가 강제하게 한다.** 모델의 자제력이 아니라 승인 모드·허용목록·권한 분리에 맡긴다.
3. **현재 시점을 확인한다.** 가격·무료 티어 같은 건 옛 정보가 쉽게 끼어든다.
4. **본업을 막는 격리는 피한다.** Local 백엔드 + 좁은 권한이, 갇힌 Docker보다 실용적일 때가 있다.

다음으로 세울 것:

- **모닝 브리핑 cron** — 간밤 미국 증시·선물·USD/KRW·뉴스 요약을 아침에 텔레그램으로. (그 전에 서버/Hermes 타임존을 `Asia/Seoul`로 맞춰야 cron이 의도한 시각에 돈다.)
- **재부팅 자동 부활 검증** — 비서의 핵심은 "항상 켜짐"이니 한 번쯤 확인.
- 신뢰가 쌓이면 좁은 행동을 승인제로 하나씩 추가, 필요하면 모델만 업그레이드.

> 핵심 한 줄: **"무엇을 할지"는 모델이 정하고, "실제 실행과 허용 여부"는 Hermes가 쥔다. 정확해야 하는 일일수록 결정론적인 층으로 내린다.**
