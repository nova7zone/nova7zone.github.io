# Oracle Cloud Free Tier — A1.Flex 확보 및 환경 구성기

> VM.Standard.E2.1.Micro 하나로 시작해서, A1.Flex까지 확보하기까지의 전 과정을 기록합니다.

---

## 0. 배경

Oracle Cloud Free Tier에는 평생 무료로 쓸 수 있는 리소스가 있다.

- **VM.Standard.E2.1.Micro** × 2개 (AMD, 1 OCPU / 1GB RAM)
- **VM.Standard.A1.Flex** (ARM, 최대 4 OCPU / 24GB RAM)
- 스토리지 200GB

E2.1.Micro 하나를 자동매매 스크립트 서버로 운영하던 중, A1.Flex의 존재를 알게 됐다.
ARM 기반에 24GB RAM이라면 Hermes Agent와 로컬 LLM 모델까지 돌릴 수 있다.
당연히 안 할 이유가 없었다.

---

## 1. 첫 번째 문제 — OCI CLI 설치

A1.Flex는 콘솔에서 직접 생성하면 `Out of host capacity` 오류가 뜬다.
자동으로 재시도하는 스크립트가 필요했고, 그러려면 OCI CLI가 먼저였다.

### apt_pkg 오류

```
ModuleNotFoundError: No module named 'apt_pkg'
```

Ubuntu 시스템의 Python 버전 불일치 문제였다.

```bash
# 확인
python3 --version         # Python 3.11
ls /usr/lib/python3/dist-packages/apt_pkg*.so
# apt_pkg.cpython-310-x86_64-linux-gnu.so
```

시스템 Python은 3.11인데 `apt_pkg`는 3.10으로 컴파일되어 있었다.
심볼릭 링크가 존재하지 않는 파일(`cpython-312-aarch64`)을 가리키고 있었던 것.

```bash
# 수정
cd /usr/lib/python3/dist-packages
sudo ln -sf apt_pkg.cpython-310-x86_64-linux-gnu.so apt_pkg.so
```

### 가상환경으로 OCI CLI 설치

시스템 Python을 건드리지 않는 가장 안전한 방법이다.

```bash
python3 -m venv ~/oci-env
source ~/oci-env/bin/activate
pip install oci-cli
pip install cffi        # _cffi_backend 누락 오류 해결
oci --version           # 3.85.0
```

스크립트에서 사용할 때는 항상 가상환경을 먼저 활성화해야 한다.

```bash
source ~/oci-env/bin/activate
```

## 2. OCI CLI 설정

### API Key 생성 (패스프레이즈 없이)

자동화 스크립트에서 매번 패스프레이즈를 입력할 수 없으므로, 패스프레이즈 없는 키를 생성한다.

```bash
openssl genrsa -out ~/.oci/oci_api_key.pem 2048
chmod 600 ~/.oci/oci_api_key.pem
openssl rsa -pubout -in ~/.oci/oci_api_key.pem -out ~/.oci/oci_api_key_public.pem

# 보안 경고 제거
echo "OCI_API_KEY" >> ~/.oci/oci_api_key.pem
```

### OCI 콘솔에 Public Key 등록

1. OCI 콘솔 → 우상단 프로필 → **My profile**
2. **Tokens and keys** 탭
3. **Add API Key** → **Paste a public key**
4. `cat ~/.oci/oci_api_key_public.pem` 내용 붙여넣기

### config 파일 설정

```ini
[DEFAULT]
user=ocid1.user.oc1..xxxxx
fingerprint=xx:xx:xx:xx:...
key_file=/home/ubuntu/.oci/oci_api_key.pem
tenancy=ocid1.tenancy.oc1..xxxxx
region=ap-chuncheon-1
```

### 연결 테스트

```bash
oci iam region list
# 패스프레이즈 없이 리전 목록이 출력되면 성공
```

## 3. 필요한 OCID 값 수집

스크립트 작성 전에 필요한 값들을 미리 확인한다.

```bash
# Availability Domain
oci iam availability-domain list
# → FGOE:AP-CHUNCHEON-1-AD-1

# Subnet OCID (현재 인스턴스가 연결된 서브넷 확인)
curl -s -H "Authorization: Bearer Oracle" http://169.254.169.254/opc/v2/vnics/
oci network vnic get --vnic-id <vnic-id>
# → subnet-id 확인

# Ubuntu 24.04 ARM 최신 이미지 OCID
oci compute image list \
  --compartment-id <tenancy-ocid> \
  --operating-system "Canonical Ubuntu" \
  --shape "VM.Standard.A1.Flex" \
  --sort-by TIMECREATED \
  --all \
  | grep -E '"id"|"display-name"'
# → Canonical-Ubuntu-24.04-aarch64-2026.04.30-1 선택
```

> **팁**: Root Compartment만 사용 중이라면 Tenancy OCID = Compartment OCID 다.

## 4. A1.Flex 확보 전략 — 단계적 확장

처음부터 4 OCPU / 24GB를 노리면 `Out of host capacity`에 막힌다.
[blog.sanguneo.com/77](https://blog.sanguneo.com/77) 에서 소개된 전략을 참고했다.

```
1 OCPU / 6GB  →  2 / 12  →  3 / 18  →  4 / 24
```

한 번에 큰 자리를 기다리는 것보다, 작은 빈자리를 여러 번 잡는 방식이 성공 확률이 높다.

## 5. 자동 재시도 스크립트

```bash
cat > ~/a1_retry.sh << 'EOF'
#!/bin/bash

source ~/oci-env/bin/activate

COMPARTMENT_ID="ocid1.tenancy.oc1..xxxxx"
SUBNET_ID="ocid1.subnet.oc1.ap-chuncheon-1.xxxxx"
IMAGE_ID="ocid1.image.oc1.ap-chuncheon-1.xxxxx"
AVAILABILITY_DOMAIN="FGOE:AP-CHUNCHEON-1-AD-1"
SSH_KEY="$(cat ~/.ssh/authorized_keys)"
OCPUS=1
MEMORY_GB=6
DISPLAY_NAME="a1-flex-instance"
LOG_FILE="$HOME/a1_retry.log"

echo "=== A1.Flex 1/6 확보 시작: $(date) ===" >> "$LOG_FILE"

while true; do
    echo "[$(date)] 생성 시도 중... (${OCPUS} OCPU / ${MEMORY_GB}GB)" >> "$LOG_FILE"

    RESULT=$(oci compute instance launch \
        --compartment-id "$COMPARTMENT_ID" \
        --availability-domain "$AVAILABILITY_DOMAIN" \
        --shape "VM.Standard.A1.Flex" \
        --shape-config "{\"ocpus\": $OCPUS, \"memoryInGBs\": $MEMORY_GB}" \
        --subnet-id "$SUBNET_ID" \
        --image-id "$IMAGE_ID" \
        --display-name "$DISPLAY_NAME" \
        --ssh-authorized-keys-file <(echo "$SSH_KEY") \
        --assign-public-ip true \
        2>&1)

    if echo "$RESULT" | grep -q '"lifecycle-state"'; then
        echo "[$(date)] ✅ 성공!" >> "$LOG_FILE"
        echo "$RESULT" >> "$LOG_FILE"
        break
    else
        echo "[$(date)] ❌ 실패 — 180초 후 재시도" >> "$LOG_FILE"
        sleep 180
    fi
done
EOF

chmod +x ~/a1_retry.sh
```

### 백그라운드 실행

```bash
nohup ~/a1_retry.sh &
tail -f ~/a1_retry.log
```

`Ctrl+C`로 로그 모니터링만 종료해도 스크립트는 계속 실행된다.

> **재시도 간격**: 180초(3분) 권장. 너무 짧으면 OCI API rate limit에 걸릴 수 있다.

## 6. 결과

**3,310번의 실패** 끝에 A1.Flex 확보에 성공했다.

- 시작: `2026년 6월 5일 (금) 16:19`
- 성공: `2026년 6월 16일 (화) 08:31`
- 소요 기간: **약 11일**
- 총 시도 횟수: **3,310회** (180초 간격)

```
=== A1.Flex 1/6 확보 시작: Fri Jun  5 16:19:29 KST 2026 ===
[Fri Jun  5 16:19:29 KST 2026] 생성 시도 중... (1 OCPU / 6GB)
...
[Tue Jun 16 08:31:18 KST 2026] ✅ 성공! 1 OCPU / 6GB 인스턴스 생성됨
```

> 춘천 리전 ARM 용량은 그만큼 경쟁이 치열하다. 포기하지 않는 것이 유일한 전략이다.

| 인스턴스 | Public IP | 스펙 | 역할 |
|---|---|---|---|
| instance-lab (E2) | 168.110.xxx.xxx | 1 OCPU / 1GB | 자동매매 스크립트 |
| a1-flex-instance (A1) | 158.179.xxx.xxx | 1→4 OCPU / 6→24GB | Hermes Agent / LLM |
