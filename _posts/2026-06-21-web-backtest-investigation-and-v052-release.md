## 들어가며

[이전 글](/posts/oci-infra-fixes-and-db-browser/)에서 6일간 dead 상태였던 웹 서비스를 복구하고 DB 조회 페이지를 만들었다. 같은 날 이어서, 다음 기능(웹 백테스트)을 위한 사전조사와 모의투자 매도 실패 분석, 그리고 릴리즈와 잔고 점검까지 진행했다.

결론부터 말하면, 웹 백테스트 기능을 위해 KIS 공식 백테스팅 엔진을 조사해 다음 세션 시작점으로 기록만 해두고, 모의투자 매도 실패 2건은 코드 분석 결과 "운영 안정성 공백 2건"으로 정리해 일단 보류했다. v0.5.2를 릴리즈한 뒤, 내일 가동을 준비하며 점검하다가 `holdings.json`과 실제 KIS 잔고가 어긋나 있는 걸 발견해 수동으로 정정했다.

## 웹 백테스트 기능 사전조사

사용자가 백테스트를 웹페이지에서 실행하고 결과를 확인하고 싶다고 요청하면서, 한국투자증권 공식 샘플 저장소의 백테스팅 엔진(`open-trading-api/backtester/`)을 참고하려는데 해당 폴더에 접근 가능한지부터 물었다.

폴더 접근이 가능함을 확인했고, QuantConnect Lean 기반에 frontend/backend가 분리된 구조(`kis_backtest/`/`examples/`/`scripts/`/`kis_mcp/` 등)와 상세 README가 있다는 것까지 확인했다. 스택 자체는 우리 FastAPI+Jinja2 대시보드와 다르지만 참고용으로는 충분했다. 사용자 지시대로 실제 구현은 시작하지 않고 `next-tasks.md`에 다음 세션 시작점으로만 기록하고 종료했다.

## 모의투자 매도 주문 실패 2건 코드 분석 (수정은 보류)

`next-tasks.md`의 "모의투자 매도 주문 실패 2건 조사 필요" 항목을 이어서 진행했다. OCI 서버 접근 권한이 없어서 실제 데이터(126640 종목의 KIS 실잔고, `cron.log` 전체 이력)는 직접 대조하지 못하고 코드 분석만 했다.

보유 종목 한도(6/5) 초과는 `strategy_v1/trading/buy.py`의 카운트 비교 로직 자체 결함이 아니라, **매도 실패의 결과**라는 것을 확인했다. 그리고 코드를 더 들여다보면서 운영 안정성 공백 2건을 발견했다.

1. `strategy_v1/trading/sell.py`에서 매도 주문이 실패해도 Telegram 알림이 없음 — 성공 시에만 `notify`를 호출하고 있었음
2. `utils/reconcile.py`가 **봇 자신의 계좌**(live_main/paper_main)는 KIS 실제 잔고 대조 없이 `holdings.json`↔`position_registry` DB(둘 다 봇 내부 상태)만 비교하고 있었음 — 추가 계좌(조회 전용)에만 실제 잔고 대조 로직이 있었음

사용자 결정으로 두 공백 모두 지금은 수정하지 않고 조사만으로 종료했다(`build-docs/update/2026-06-21-paper-sell-failure-investigation.md`에 상세 기록). 코드 변경은 없었다.

> 두 번째 공백("봇 자신의 계좌는 KIS 실제 잔고와 대조하는 장치가 없다")은 이날 처음 명시적으로 기록됐는데, 이틀 뒤(6/23) 본계좌 실시간 잔고 동기화 기능으로 해결된다 — 그 사이 6/22에 더 큰 사고를 한 번 겪고서야 다시 수정 우선순위에 오르게 된다.

## PR #51 merge + v0.5.2 릴리즈 게시

PR #51(웹 백테스트 기록 + 모의투자 매도 실패 조사 결과)을 생성하고 CI(pytest) 통과를 확인한 뒤 사용자가 GitHub에서 직접 머지했다.

`release-notes` skill 절차에 따라 v0.5.1 이후 머지된 PR 4건(#48~#51)을 조사해 `build-docs/release-notes-draft-v0.5.2.md` 초안을 작성했다. 다음 버전 번호는 사용자에게 확인받아 `v0.5.2`(patch)로 결정했다 — DB 조회 페이지(#50)는 관리자용 보조 기능으로 보고 minor 대신 patch를 선택했다.

`pytest tests/` 575개 전체 통과를 재확인한 후 `v0.5.2` 태그를 생성·push하고 GitHub Release를 게시했다. 사용자가 따로 "DB 조회 페이지 OCI 배포 후 실사용 확인"을 직접 완료했다고 알려와 `next-tasks.md`의 해당 항목도 종료 처리했다.

## ❌ holdings.json ↔ KIS 실제 잔고 불일치 발견·정정

v0.5.2 OCI 배포 후 "내일 정상 가동 준비됐는지" 확인 요청을 받고 git/dry-run/pytest/crontab/`kis-web.service`를 순차 점검했는데, 전부 정상이었다. 그런데 점검 중 `data/holdings.json`의 6종목이 KIS 실제 모의투자 잔고에 전혀 없고, `holdings.json`에는 없는 027740(마니커) 148주가 실제로 보유 중인 것을 `get_balance()` 직접 호출로 발견했다. DB 조회 결과 027740은 6/12에 매수·매도 모두 체결 확인을 거쳐 종료된 거래였다.

`api/kis_api.py::check_order_status()` 코드를 다시 읽어보니 CLAUDE.md의 "미체결 목록에 없으면 체결 완료로 간주"라는 기존 설명이 stale했다는 것도 확인했다 — 실제로는 체결내역 재조회까지 거치는 안전한 로직이라 코드 버그는 아니었고, **KIS 모의투자 계좌의 주기적 리셋**으로 추정했다. CLAUDE.md "알려진 주의사항"을 정정하고 리셋 추정 사실을 추가했다.

OCI에서 `holdings.json`을 027740 기준으로 직접 교정(백업 후 덮어쓰기, git 비대상)하고 `python runner.py --reconcile`로 `position_registry` DB도 동기화(6종목 sold/manual, 027740 holding 삽입)했다. 코드 변경은 없었다(문서 정정만, 데이터 정정은 OCI 서버에서 직접 수행).

> 이때는 "모의투자 계좌의 주기적 리셋"으로 결론 내렸지만, 다음 날(6/22)에 거의 똑같은 모양의 불일치가 다시 나타나면서 진짜 원인이 따로 있다는 게 드러난다.
